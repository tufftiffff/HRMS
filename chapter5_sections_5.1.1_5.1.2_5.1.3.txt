CHAPTER 5: IMPLEMENTATION AND TESTING

5.1.1 Server Implementation (MQTT Protocol)
-------------------------------------------
Purpose
- The server side functions as the central controller of the system. It is responsible for establishing a stable connection to the MQTT Broker, receiving data from clients through subscriptions, processing the incoming payloads based on application rules, and optionally publishing acknowledgments or commands back to clients.

Connection and Configuration
- Broker URL: The server connects to the MQTT Broker using a broker endpoint such as tcp://<broker-host>:1883 (standard TCP) or ssl://<broker-host>:8883 (TLS encrypted).
- Client ID: A unique identifier is assigned to the server (e.g., "Server") so the Broker can manage sessions and message delivery correctly.
- Authentication: If enabled, the server connects using a username/password. For production-grade deployments, TLS certificates and broker-side access control are recommended.
- Quality of Service (QoS): QoS determines delivery guarantees:
  - QoS 0: at most once (fast, no guarantee)
  - QoS 1: at least once (common for important events)
  - QoS 2: exactly once (highest reliability, more overhead)
- Session and Persistence:
  - Clean session ON: no persisted subscriptions/queued messages after disconnect
  - Clean session OFF: enables persistent subscriptions and delivery of queued messages (depends on broker settings)

Server Processing Workflow
1) Connect to MQTT Broker:
   - Initialize MQTT client instance
   - Configure broker URL, client ID, credentials (if any), keep-alive, and reconnect behavior
2) Subscribe to topics:
   - Subscribe to one or more topics that clients publish to (e.g., "<project>/events/in", "<project>/attendance/in", or a wildcard like "<project>/sensor/#")
3) Receive and validate messages:
   - Decode payload (commonly JSON)
   - Validate required fields (e.g., deviceId/userId/timestamp/type/value)
   - Reject malformed/duplicate/out-of-window timestamps as needed
4) Execute business logic:
   - Map payload to system entities (attendance record, log entry, sensor reading)
   - Apply rules such as deduplication, threshold checks, or routing to other services
5) Store and log:
   - Save relevant records to the database for audit and reporting
   - Log receive attempts, validation failures, and successful processing results
6) Publish responses (optional):
   - Publish acknowledgments (".../ack") so clients know data was accepted
   - Publish commands/config (".../cmd") if the server controls client behavior

Reliability and Fault Handling
- Keep-alive and auto-reconnect help the server recover after network disruptions.
- Last Will and Testament (LWT) can be configured to publish an "offline" status if the server disconnects unexpectedly.
- Broker-side topic permissions (ACL) reduce the risk of unauthorized publish/subscribe operations.


5.1.2 Client Implementation (MQTT Protocol)
-------------------------------------------
Purpose
- The client side represents the data source (device/app) that produces events or readings and publishes them to the MQTT Broker. The client may also subscribe to command/config topics to receive instructions from the server.

Client Connection and Configuration
- Broker URL: Clients connect to the same Broker endpoint used by the server.
- Client ID: Each client uses a unique client ID, typically derived from a device identifier (e.g., "Client-<deviceId>") to avoid collisions.
- Authentication: Clients use broker credentials if required. In secure deployments, TLS is used to encrypt traffic and verify broker identity.
- QoS: Important client events (e.g., attendance submission) typically use QoS 1 so messages are delivered reliably even under unstable networks.

Client Processing Workflow
1) Connect to MQTT Broker:
   - Configure broker endpoint, client ID, credentials, keep-alive, and reconnect strategy
2) Publish data:
   - Publish event-driven messages (e.g., attendance check-in, status changes) or periodic readings (e.g., sensor values)
   - Use a clear topic structure so the server can route and process data correctly
3) Subscribe to server topics (optional):
   - Subscribe to command/config topics (e.g., "<project>/client/<id>/cmd") if the server sends instructions
4) Handle commands (optional):
   - Validate command payload and apply safe operations (e.g., update interval, request resend)
5) Offline resilience (if required):
   - Auto-reconnect on disconnect
   - Retry publishing with backoff
   - Optional local buffering if the system requires guaranteed delivery during outages

Typical Payload Format (example structure)
- JSON payload commonly contains:
  - deviceId / clientId
  - userId (if applicable)
  - timestamp (ISO format or epoch)
  - event type (e.g., "CHECK_IN", "STATUS", "SENSOR_READING")
  - value(s) or additional metadata (location, signal strength, etc.)

Client Testing Considerations
- Verify messages published by the client are received by the server subscription.
- Verify server acknowledgments or commands are received by the client (if implemented).
- Simulate disconnect/reconnect to confirm reliability and correct session behavior.


5.1.3 Database Configuration
----------------------------
Purpose
- Database configuration provides persistent storage for system operations such as event logs, attendance records, device/client activity, and reporting. A properly configured database supports data integrity, efficient queries, and secure access controls.

Database Environment
- Database engine: Common choices include MySQL/MariaDB (often used with XAMPP stacks).
- Host: Localhost (127.0.0.1) for local deployment or a LAN/hosted IP if the database is hosted on another server.
- Port: Default MySQL port is 3306 (unless customized).
- Database name: A dedicated schema for the project (e.g., "hrms" or "iot_db") separates application data cleanly.

Connection Parameters (report-ready)
- Host: <db_host>  (e.g., 127.0.0.1)
- Port: <db_port>  (e.g., 3306)
- Database: <db_name>
- Username: <db_user>
- Password: <db_password>
- Charset/Collation: utf8mb4 recommended for full Unicode support

Data Storage and Structure (typical)
- Tables commonly include:
  - Event logs: raw incoming message metadata (topic, timestamp, status)
  - Attendance records: userId, check-in time, method/source, status
  - Device/client registry: deviceId, last_seen, state, notes (optional)
- Indexing:
  - Index frequently queried fields (e.g., userId, deviceId, created_at) for reporting speed.
- Integrity:
  - Use foreign keys where appropriate (e.g., attendance.user_id -> users.id).

Database Security and Protection
- Least privilege:
  - The application database user should only have the required permissions (e.g., SELECT/INSERT/UPDATE) rather than full admin rights.
- Safe queries:
  - Use parameterized queries / ORM to prevent SQL injection and ensure consistent data handling.
- Network restrictions:
  - Do not expose the database publicly. Restrict to the application server network (firewall/VPC rules).
- Backups:
  - Implement regular backups and test restoration to ensure data recoverability.


Setting for Server, Network Protocol, IP, Database, Security (Detail Description)
-------------------------------------------------------------------------------
Server
- The server runs as the main processing component. It maintains a persistent MQTT session with the Broker, subscribes to inbound topics, validates and processes payloads, stores records in the database, and publishes acknowledgments/commands when required.

Network Protocol
- MQTT over TCP:
  - Commonly used for local networks and testing environments.
- MQTT over TLS (recommended for production):
  - Encrypts data in transit, protects credentials, and prevents interception and tampering.

IP / Addressing
- Broker Address:
  - Local testing: 127.0.0.1
  - LAN deployment: 192.168.x.x (or a resolvable hostname)
  - Hosted deployment: public hostname with TLS enabled
- The chosen address must be reachable by both server and all clients, and firewall rules must allow the broker port.

Database
- The database can be hosted on the same machine as the server (simple deployment) or on a separate host (scalable deployment).
- Use a dedicated schema, controlled credentials, and proper indexing for efficient reporting.

Security
- MQTT:
  - Enable authentication (username/password).
  - Apply broker ACL rules to restrict which clients can publish/subscribe to which topics.
  - Prefer TLS for encryption in transit.
- Application:
  - Enforce authentication and role-based access control for dashboards and reports.
  - Log all receive/processing attempts for auditability.
- Database:
  - Protect credentials, restrict network access, and apply least privilege permissions.
